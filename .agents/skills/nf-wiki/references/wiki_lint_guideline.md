# Wiki Lint & Synchronization Guideline (v1.0)

이 지침은 `nf-wiki` 스킬의 **[린트 모드]** 수행 시 Antigravity가 준수해야 할 상세 작업 절차입니다. 지식의 부패를 막고 코드와 위키 간의 '제로 갭(Zero-Gap)'을 달성하는 것을 목적으로 합니다.

## Phase 1. 물리적 동기화 (Codebase Sync)
지식의 토대인 프로젝트의 물리적 상태와 위키의 기술 정보를 일치시키는 단계입니다.

### 1.1. 구조 및 지도 점검 (`codebase_map.md`)
* **파일 트리 대조**: 실제 터미널의 `tree` 결과와 `codebase_map.md`의 구조가 일치하는지 전수 조사합니다.
* **상태 값 현행화**: 완료(✅), 진행 중(🔄), 미착수(⬜) 마커가 현재 작업 진척도와 일치하는지 확인합니다.
* **서브 프로젝트 정렬**: `parent_wiki`와 각 `p{n}_wiki` 간의 폴더 경계가 명확히 구분되어 있는지 점검합니다.

### 1.2. 가비지(Garbage) 검토
* 가비지 후보군(Garbage Candidates) 판정: 다음 기준에 따라 프로젝트에 불필요한 파일을 식별합니다.
    * 기능적 필수성: 프로젝트가 빌드되거나 실행되는 데 반드시 필요한 소스 코드, 설정 파일인가?
    * 운영적 필요성: 초기화(Setup) 과정이나 지속적인 운영/유지보수에 사용되는 스크립트인가?
    * 지식적 가치 연결: 운영 관련 Runbook이나 **연구 성과/실험 문서(experiments.md)**에서 참조하거나 링크하고 있는 데이터/문서인가?
    * 위 세 가지 중 어디에도 해당하지 않는 임시 파일, 테스트 잔재, 중복 백업 등은 '가비지 대상'으로 분류합니다.

### 1.3. 가비지 처리 및 반영
* 가비지 리스트 작성: 식별된 파일들을 `00_schema/garbage_list.md` 또는 `codebase_map.md` 내 '정리 대상' 섹션에 일시적으로 기록합니다.
* 지식 보존 절차: 삭제 전, 해당 파일에 담긴 유의미한 로직이나 데이터가 있다면 `interfaces/`나 `decisions/`로 지식을 이관했는지 확인합니다.
* 물리적 격리/삭제: 사용자 컨펌 후 파일을 삭제하거나, `archive/` 폴더로 이동시켜 `codebase_map.md`를 실제 물리적 상태와 동기화합니다.

### 1.4. 환경 인프라 점검 (`environment.md`)
* **런타임 대조**: 현재 가상환경의 패키지 버전(`pip freeze` 등)과 `environment.md`에 명시된 버전이 일치하는지 확인합니다.
* **인프라 변경 감지**: Dockerfile, requirements.txt, poetry.lock, .env 파일 등의 변경 사항이 위키에 반영되었는지 검토합니다.
* **개인정보 보안**: .env 파일 에서 개인정보 관련 내용은 절대 위키에 반영하지 않습니다. 대신, 해당 환경 변수는 `os.environ`으로 불러오는 방식으로 위키에 기록합니다. (예: `API_KEY = os.environ['API_KEY']`)

## Phase 2. 지식 완전성 인증 (Knowledge Coverage)
코드베이스에 존재하는 로직이 위키에 누락 없이 '원자화'되어 있는지 확인하는 단계입니다.

### 2.1. 인터페이스 누락 전수 조사 (`interfaces/`)
* **코드 스캔**: 코드 내의 신규 함수, 클래스, API 엔드포인트를 추출합니다.
* **엔티티 매핑**: 추출된 각 인터페이스가 `interfaces/` 폴더 내에 개별 문서로 존재치 않는 경우, 즉시 신규 문서를 생성합니다.
* **디테일 검증**: 원본 코드의 타입 힌트와 위키의 명세가 요약 없이 1:1로 매칭되는지 확인합니다.

### 2.2. 미등록 이슈 및 결정사항 감사 (`errors/`, `decisions/`)
* **로그 및 히스토리 분석**: 최근의 커밋 메시지나 에러 로그를 분석하여, 해결되었으나 `errors/`에 등록되지 않은 사례를 찾아냅니다.
* **결정 근거 확인**: 코드 내 주석으로만 남겨진 "Why"에 대한 기술적 판단을 `decisions/` 문서로 격상하여 기록합니다.

> **[인증]**: 위 단계가 완료되면 "코드베이스의 모든 물리적/기술적 자산이 위키에 원자 단위로 존재함"을 인증합니다.

## Phase 3. 논리적 정합성 및 연결 (Logic & Links)
위키 내부에 축적된 지식 간의 모순을 해결하고 복리 효과를 위한 연결망을 구축하는 단계입니다.

### 3.1. 상호 참조 및 MoC 점검 (`index.md`)
* **Broken Link 검사**: `[[...]]` 형식의 링크 중 존재하지 않는 파일을 가리키거나 경로가 잘못된 것을 수정합니다.
* **MoC 요약 동기화**: `00_schema/index.md`에 기재된 각 엔티티의 한 줄 요약이 실제 파일의 최신 내용과 일치하는지 갱신합니다.

### 3.2. 정보 모순 식별 (Consistency Check)
* **충돌 발견**: 예를 들어, `interfaces/`에서는 특정 변수를 `int`로 정의했으나, `errors/` 문서에서는 해당 변수의 `string` 처리 문제로 에러가 발생했다고 기록된 경우 등을 식별합니다.
* **논리적 통합**: 모순이 발견되면 최신 코드 기준으로 정보를 통합하고, `log.md`에 'Consistency Resolved'로 기록합니다.

## Phase 4. 결과 보고 및 마감
* **Lint 리포트**: "Sync 완료, 가비지 X건 제거(또는 이동), 신규 엔티티 X개 생성, 모순 Y건 해결"의 형태로 사용자에게 보고합니다.
* **다음 액션 제안**: 지식의 공백이 발견된 부분(예: "데이터 파이프라인 최적화 결정 사항이 모호함")에 대해 추가 조사를 제안합니다.
