# nf-task-planner

PRD를 세션 단위 구현 Task로 분해하고 `p{n}_pjt_tasks.md`를 생성하는 스킬입니다.

> 산출물 템플릿은 `references/output-template.md`를 참조하세요.

---

## § 1. 역할 및 입출력

### 경로 규칙 — 반드시 먼저 확인

작업 시작 전 `docs/` 폴더 하위 디렉토리명을 확인하여 실제 존재하는 경로를 사용합니다.
현재 프로젝트 구조:

```
docs/
├── p1_shared/      → p1_shared_PRD.md 등
├── p2_kdms/        → p2_kdms_PRD.md 등
├── p3_usdms/       → p3_usdms_PRD.md 등
├── p4_manager/     → p4_manager_PRD.md 등
└── parent/         → 상위 PRD
```

경로 패턴: `docs/{서브프로젝트 폴더명}/` (예: `docs/p2_kdms/`)
산출물 파일명 패턴: `{서브프로젝트 폴더명}_pjt_tasks.md` (예: `p2_kdms_pjt_tasks.md`)

> 새로운 서브프로젝트가 추가될 경우 `docs/` 하위에 새 폴더가 생기며,
> 이 스킬은 그 폴더명을 그대로 따릅니다. 가정하지 말고 실제 폴더를 확인하세요.

**입력:**
- `docs/{서브프로젝트폴더}/{서브프로젝트폴더}_PRD.md` (필수, 예: `docs/p2_kdms/p2_kdms_PRD.md`)
- `docs/{서브프로젝트폴더}/{서브프로젝트폴더}_TRD.md` (선택)
- `docs/{서브프로젝트폴더}/{서브프로젝트폴더}_DB_SCHEMA.md` (선택)

**출력:**
- `docs/{서브프로젝트폴더}/{서브프로젝트폴더}_pjt_tasks.md` (단일 산출물)
- 예시: `docs/p2_kdms/p2_kdms_pjt_tasks.md`

**이 스킬의 책임 범위 밖:**
- Task 상세 명세 작성 → `nf-task-spec-writer` 스킬
- 실제 구현 → 구현 Agent

## § 2. Task 크기 기준 — 세션 단위 분할

**하나의 Task = 한 AI 세션 내에 완결 가능한 논리적 독립 구현 단위**

크기 판단 기준은 구현 복잡도와 예상 토큰 소비량입니다.

**적절한 Task 조건 (모두 충족):**
- 단일 세션 내 구현·테스트·커밋까지 완결 가능
- 다른 미완성 Task 없이 독립적으로 동작 검증 가능
- 하나의 명확한 기능적 목적을 가짐
- 구현 대상 파일 목표 10개 미만

**분할 신호:**
- 여러 계층(데이터 → 로직 → API → UI)에 걸쳐 있음
- 독립적으로 테스트 가능한 기능이 둘 이상 혼재
- 외부 API 연동 + 알고리즘 + 동시성 처리 등 고위험 영역 혼재

**통합 신호:**
- 단독으로는 동작 검증이 불가능한 단편 로직
- 독립 테스트가 불가능한 유틸 함수 하나

**분할 방향: 기능 단위 End-to-End (세로 분할)**

```
올바른 분할: 기능 단위
"포트폴리오 생성" = 입력 검증 + 저장 + 응답 → 독립 동작 가능

잘못된 분할: 계층 단위 (수평 분할)
"전체 DB 저장 로직" → 단독 검증 불가
```

## § 3. 작업 프로세스

```
Step 1: PRD 전체 요구사항 파악
         └─ 기능 목록 추출 (TRD·DB_SCHEMA 있으면 기술 제약 병행 파악)
Step 2: Phase 구분 (MVP vs 확장)
Step 3: Task 분해 (§ 2 기준 적용)
Step 4: 의존성 분석 및 구현 순서 확정
Step 5: PRD 요구사항 커버리지 검증 (§ 5 참조)
Step 6: 초안 사용자 검토 요청
Step 7: p{n}_pjt_tasks.md 최종 작성 (references/output-template.md 형식 사용)
Step 8: 최종 승인
```

**Step 6 검토 요청 시 반드시 포함할 내용:**

```
[Task 분해 초안]
- Phase 1 (MVP): N개 Task
- Phase 2 (확장): N개 Task
- 총 Task: N개

[주요 판단 확인]
1. {분할/통합 결정 사항 및 이유}
2. {의존성 순서 주요 결정}

[검토 사항]
- Phase 구분이 PRD 우선순위와 일치하는지?
- Task 크기 조정이 필요한 항목?
```

## § 4. 의존성 분석

| 타입 | 의미 | 표기 |
|---|---|---|
| 순차 | Task A 완료 후 B 시작 가능 | `의존: T-XXX` |
| 병렬 | 동시 진행 가능 | 의존성 없음 |
| 통합 | 복수 Task 완료 후 통합 | `의존: T-XXX, T-YYY` |

**원칙:**
- 공통 기반(DB 설정, 공유 유틸 등)은 최우선 Task로 배치
- 순환 의존성 발견 시 → 공통 부분을 별도 선행 Task로 추출
- 의존성 체인이 3단계 이상이면 중간 지점에서 분할 검토

---

## § 5. PRD 요구사항 커버리지 검증

Task 분해 완료 후 반드시 수행합니다.

1. PRD에서 기능 요구사항 목록을 항목별로 추출
2. 각 요구사항이 어느 Task에서 구현되는지 매핑
3. 매핑되지 않은 요구사항 식별 → Task 추가 또는 기존 Task 범위 조정
4. 검증 결과를 산출물의 § 4 커버리지 섹션에 기록

커버리지 표는 `references/output-template.md`의 § 4 형식을 사용합니다.

---

## § 6. 완료 기준 체크리스트

작업 완료 전 다음을 모두 확인합니다.

**Task 품질:**
- [ ] 모든 Task가 한 세션 내 완결 가능한 단위인가?
- [ ] 모든 Task가 독립적으로 동작 검증 가능한가?
- [ ] Task명만으로 구현 목적이 명확한가?
- [ ] 구현 범위 요약이 PRD 요구사항에 직접 대응하는가?

**커버리지:**
- [ ] PRD 전체 요구사항이 § 4 커버리지 표에 매핑되었는가?
- [ ] 누락된 요구사항이 없거나, 누락 사유가 명시되었는가?

**계획 정합성:**
- [ ] 의존성 흐름에 순환이 없는가?
- [ ] Phase 구분이 PRD 우선순위와 일치하는가?
- [ ] 사용자 최종 승인 완료

**완료 후 전달 형식:**

```
[산출물]
✅ docs/{서브프로젝트폴더}/{서브프로젝트폴더}_pjt_tasks.md
   (예: docs/p2_kdms/p2_kdms_pjt_tasks.md)

[요약]
- 총 N개 Task (Phase 1: N개 / Phase 2: N개)
- PRD 요구사항 N개 전체 커버
- 미커버: {없음 또는 항목 및 사유}

다음 단계: nf-task-spec-writer 스킬로 구현 직전 Task별 상세 명세 작성
```
