# 개발자 작업 수칙 및 가이드라인 (Developer Guidelines)

## 0. 개요 (Overview)
본 문서는 프로젝트(`01_usdms`)의 일관성, 안정성, 그리고 사용자와의 신뢰 유지를 위해 모든 AI 에이전트 및 개발자가 반드시 준수해야 할 작업 수칙을 정의합니다. 
**새로운 세션을 시작할 때 이 문서를 가장 먼저 숙지하십시오.**

---

## 1. 환경 및 실행 (Environment & Execution) (Critical)

### 1-1. 가상환경 필수 사용
*   **원칙:** 모든 Python 스크립트 실행은 반드시 `usdms_env` 가상환경 내에서 이루어져야 합니다.
*   **금지:** 시스템 글로벌 환경(`/usr/bin/python3`) 사용 금지.
*   **방법:** 
    *   터미널 명령 시: `conda run -n usdms_env python <script>` 또는 `source activate usdms_env && python <script>` 명시.
    *   *주의:* 에이전트 도구(`run_command`)는 상태를 저장하지 않으므로, 매 명령마다 환경을 활성화하거나 절대 경로를 사용해야 합니다.

### 1-2. 의존성 관리 (.env 및 패키지)
*   **원칙:** `.env` 파일은 DB 접속 정보 등 민감 정보를 포함하므로, 코드 내에서 `load_dotenv()`를 통해 안전하게 로드합니다.
*   **금지:** 코드 내에 비밀번호 하드코딩 절대 금지.
*   **설치 (uv 우선):** 패키지 설치 시 **`uv pip install`을 기본으로 사용**합니다. `uv` 사용이 불가능한 예외적인 경우에만 `pip install`을 제안하며, 이때 `uv`로 설치하지 못하는 이유를 정확히 안내해야 합니다. (무단 패키지 추가 금지)

### 1-3. 터미널 명령 실행 권한 (Strict)
*   **원칙:** **터미널 명령어(run_command)를 에이전트가 단독으로 실행하지 마십시오.** (검색/조회 용도 제외)
*   **변경/실행:** DB 데이터 변경(UPDATE/DELETE), 파일 삭제, 파이썬 스크립트 실행 등 **State를 변경하는 모든 명령**은 계획 단계에서 승인을 받은 후, **사용자에게 명령어를 제안하여 직접 실행**하도록 유도하거나, 명시적인 실행 허가를 득한 후에만 수행합니다.
*   **배경:** 에이전트의 자동 실행이 의도치 않은 결과(잘못된 삭제, 무한 루프 등)를 초래할 수 있기에, 사용자의 통제권이 최우선입니다.

## 2. 작업 워크플로우 (Workflow)

### 2-1. 선(先) 계획, 후(後) 실행
*   **원칙:** 코드를 수정하기 전, 반드시 **수정 계획(Implementation Plan)**을 작성하여 사용자에게 보고(Notify)합니다.
*   **내용:** 문제의 원인, 수정 방안, 예상되는 사이드 이펙트를 명시해야 합니다.
*   **승인:** 사용자의 명시적인 승인(Yes/Proceed)이 떨어지기 전에는 파일 수정 도구(`replace_file_content` 등)를 호출하지 않습니다.

### 2-2. 논리적 검증 우선
*   **원칙:** "이럴 것이다"라는 추측으로 코드를 수정하지 않습니다. (예: 데이터가 없다고 해서 바로 로직 버그로 단정 짓지 말 것)
*   **행동:** 로그 분석(grep), DB 조회(SELECT), 미니 테스트 스크립트 실행 등을 통해 **팩트(Fact)**를 먼저 확보한 뒤 수정 계획을 수립합니다.

---

## 3. 문서화 표준 (Documentation Standards)

### 3-1. 상세함의 수준 (Granularity)
*   **원칙:** 문서는 코드를 읽지 않아도 로직의 흐름, 분기 조건, 입출력 테이블을 파악할 수 있을 정도로 상세해야 합니다.
*   **내용:** 
    *   단순히 "데이터를 저장한다"가 아니라, "어떤 테이블의 어떤 컬럼을 기준으로 중복을 체크하고(Upsert), 충돌 시 어떤 전략을 쓰는지" 기술해야 합니다.
    *   알고리즘의 한계(Known Constraints)나 예외 처리 로직도 반드시 포함합니다.

### 3-2. 언어 (Language)
*   **원칙:** 모든 문서 및 사용자와의 대화는 **한국어(Korean)**를 기본으로 합니다. (코드 주석 및 변수명은 영어 허용)

---

## 4. 데이터베이스 및 스키마 (Database)

### 4-1. 기존 자원 활용 (Conservation)
*   **원칙:** 새로운 문제를 해결하기 위해 무조건 테이블을 추가하거나 컬럼을 늘리는 것은 지양합니다.
*   **분석:** 기존 테이블(`updated_at`, `log` 등)을 활용할 수 있는지 깊이 있게 고민하고, 불가피한 경우에만 스키마 변경을 제안합니다.

### 4-2. 노이즈 데이터 관리
*   **원칙:** 분석/백테스팅에 방해가 되는 노이즈(Noise) 데이터(예: 장중 일시적 티커 변경, 빈 깡통 데이터)는 보존하기보다 과감히 필터링/삭제하는 것이 원칙입니다. (Clean Data Policy)
