## 1. 언어 규칙 (Language Protocol)
- 모든 답변과 설명은 항상 **한국어**로 작성한다.

## 2. 문서 무결성 보호 및 도구 사용 원칙 (Integrity & Tool Usage)
- **에이전트 도구 우선순위**: 파일 읽기 및 수정 시 반드시 에이전트 전용 도구(`view_file`, `replace_file_content`, `multi_replace_file_content`)를 최우선으로 사용한다.
- **터미널 명령(sed 등) 금지**: 문서 수정 시 `sed`, `echo` 등의 터미널 명령을 사용하여 직접 파일을 편집하는 것을 엄격히 금지한다. 이는 인코딩 깨짐이나 예상치 못한 데이터 손상을 방지하기 위함이다.
- **손상된 문서 대응**: `view_file`이 "알 수 없는 형식(Binary)" 등으로 파일을 읽지 못할 경우:
    1. 절대 터미널 명령으로 강제 수정을 시도하지 않는다.
    2. 즉시 사용자에게 보고하고, 파일의 인코딩 상태나 특수 문자 포함 여부를 확인하여 조언을 구한다.
    3. 문서 인식을 방해하는 특수 문자가 발견되면 이를 '금지어' 또는 '주의 문자'로 기록하여 향후 문서 작성 시 사용하지 않도록 관리한다.

## 3. 작업 절차 및 검증 (Workflow & Verification)
- **사전 분석 필수**: 전역 치환(Global Replace)이나 대규모 수정을 수행하기 전, 반드시 전체 파일 내용을 읽거나 `grep_search` 등을 통해 현재 상태를 정확히 파악한다. (이미 수정된 상태인지, 중복 접두사가 발생할 위험이 없는지 확인)
- **단계적 수정**: 복잡한 수정은 한 번에 처리하기보다 논리적 단위로 나누어 수행하며, `multi_replace_file_content`를 사용하여 컨텍스트를 유지한다.
- **사후 검증**: 작업 완료 후에는 수정된 영역을 다시 읽어(`view_file`) 의도한 대로 정확히 수정되었는지, 문서 구조(표, 코드 블록 등)가 깨지지 않았는지 반드시 확인한다.

## 4. 커뮤니케이션 가이드라인 (Communication)
- 작업의 위험도가 높거나 의도가 불분명한 경우, 독단적으로 진행하기보다 사용자에게 예상되는 결과나 위험을 설명하고 확인을 받은 후 진행한다.
- 터미널 명령을 사용해야만 하는 특수한 상황(예: 인코딩 복구 등)이 발생하면, 반드시 그 이유와 실행할 명령어를 사용자에게 상세히 설명하고 승인을 득한다.
