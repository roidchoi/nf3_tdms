# Skill Creator

새로운 스킬을 생성하고 반복적으로 개선하기 위한 가이드입니다.

전체적인 프로세스는 다음과 같습니다:
1. 사용자의 의도를 파악하고 스킬의 대략적인 기능과 구조를 결정합니다.
2. 스킬의 초안을 작성합니다. (경로 분리 정책에 따라 배포 경로 결정)
3. 테스트 프롬프트를 만들고 시뮬레이션(에발, Eval)을 진행합니다.
4. 평가 결과를 통해 스킬을 개선하고, 이 과정을 반복합니다.
5. 스킬의 설명을 대화형으로 최적화하여 트리거 정확도를 높입니다.
6. 최종 스킬을 사용자 환경(Workflows 또는 Skills)에 등록합니다.

---

## 1. 스킬 생성 및 경로 결정 (Creating a Skill)

### 의도 파악 (Capture Intent)

먼저 사용자의 의도를 파악하는 것으로 시작합니다. 현재 대화 내용에 사용자가 스킬로 만들고 싶어 하는 워크플로우가 이미 포함되어 있을 수 있습니다(예: "이걸 스킬로 만들어줘"라고 요청하는 경우). 그렇다면 대화 기록에서 먼저 해답을 추출하세요. 사용된 도구들, 단계의 순서, 사용자가 수정했던 내용, 입력/출력 포맷 등을 파악합니다. 빈틈이 있다면 사용자에게 질문하여 채우고, 다음 단계로 넘어가기 전에 사용자에게 확인을 받아야 합니다.

1. 이 스킬이 에이전트(나)로 하여금 무엇을 할 수 있게 해야 합니까?
2. 이 스킬은 언제 트리거되어야 합니까? (사용자의 어떤 문구/상황에서)
3. 예상되는 출력 형식은 무엇입니까?
4. 스킬이 잘 작동하는지 확인하기 위해 테스트 케이스를 설정해야 할까요? 결과물을 객관적으로 검증할 수 있는 스킬(파일 변환, 데이터 추출, 코드 생성, 정해진 워크플로우 단계 등)은 테스트 케이스가 큰 도움이 됩니다. 주관적인 결과물(글쓰기 스타일, 아트 등)을 내는 스킬은 종종 테스트 케이스가 필요하지 않을 수 있습니다. 스킬 유형에 따라 적절한 기본 방향을 제안하되, 최종 결정은 사용자에게 맡기세요.

### 인터뷰 및 조사 (Interview and Research)

엣지 케이스(예외 상황), 입력/출력 포맷, 예시 파일, 성공 기준, 종속성 등에 대해 선제적으로 질문하세요. 이 부분이 완전히 정리될 때까지 테스트 프롬프트 작성은 보류합니다.

사용 가능한 MCP(Model Context Protocol)가 있는지 확인하세요. 만약 조사(문서 검색, 유사한 스킬 찾기, 모범 사례 검색 등)에 유용하다면 MCP를 활용해 문맥을 파악한 뒤 사용자에게 접근하세요. 사용자의 부담을 줄이기 위해 미리 컨텍스트를 준비하는 것이 좋습니다.

### SKILL.md 작성 (Write the SKILL.md)

사용자와의 인터뷰를 바탕으로 다음 구성 요소를 채웁니다:

- **name**: 스킬의 고유 식별자.
- **description**: 언제 트리거될지, 무엇을 하는지 적습니다. 이 필드는 스킬이 발동되는 핵심 메커니즘입니다. 스킬이 하는 일뿐만 아니라 이 스킬을 언제 사용해야 하는지에 대한 '구체적인 컨텍스트'를 반드시 포함하세요. 모든 "언제 사용할 것인가"에 대한 정보는 본문이 아닌 이곳에 들어가야 합니다. 
  *(참고: 에이전트는 종종 스킬이 유용함에도 불구하고 호출하지 않는 '과소 트리거' 경향이 있습니다. 이를 방지하기 위해 스킬 설명란을 다소 '적극적(pushy)'으로 작성하세요. 예를 들어, 단순히 "사내 데이터를 표시하는 간단한 대시보드를 구축하는 방법"이라고 적는 대신 "사내 데이터를 표시하는 간단한 대시보드를 구축하는 방법. 사용자가 대시보드, 데이터 시각화, 내부 지표를 언급하거나, 명시적으로 '대시보드'를 요구하지 않더라도 회사 데이터를 표시하고 싶어할 때마다 이 스킬을 반드시 사용하도록 하십시오."라고 적으세요.)*
- **compatibility**: 필요한 도구, 종속성 (선택 사항, 거의 필요하지 않음).
- **스킬 본문(the rest of the skill)**: 나머지 마크다운 지침들.

### 스킬 작성 가이드 (Skill Writing Guide)

#### 스킬 구조 및 배포 정책 (Anatomy of a Skill)

스킬의 복잡도에 따라 다음 두 가지 형태 중 하나로 작성합니다:

**A. 워크플로우 (단일 Markdown)**
단일 `.md` 파일로 동작하는 경우:
```
.agents/workflows/
└── skill-name.md (YAML 프론트매터 및 마크다운 지침 포함)
```

**B. 복합 스킬 (디렉토리 구조)**
스크립트 등 부가 파일이 필요한 경우:
```
.agents/skills/skill-name/
├── SKILL.md (필수, YAML 프론트매터 포함)
└── [선택적 리소스들]
    ├── scripts/    - 반복적이거나 결정론적인 작업을 위한 실행 코드
    ├── references/ - 필요 시 컨텍스트에 로드할 참고 문서
    └── assets/     - 출력에 사용되는 파일들 (템플릿, 아이콘 등)
```

#### 점진적 정보 제공 (Progressive Disclosure)

스킬은 3단계 로딩 시스템을 사용한다고 생각하고 디자인하세요:
1. **Metadata (name + description)**: 항상 컨텍스트에 상주함 (~100단어)
2. **SKILL.md 본문**: 스킬이 트리거될 때 컨텍스트에 로드됨 (500줄 이하 권장)
3. **번들 리소스(Bundled resources)**: 필요할 때만 로드됨 (제한 없음, 스크립트는 로딩 없이 실행 가능)

**핵심 패턴:**
- `SKILL.md`는 가급적 500줄 이내로 유지하세요. 이 한계에 도달하고 있다면 추가적인 계층(references 등)을 만들고, 에이전트가 다음으로 어디를 확인해야 할지 명확한 포인터를 제공하세요.
- `SKILL.md`에서 참조 파일들을 명확히 언급하고, 언제 그 파일들을 읽어야 하는지 안내하세요.
- 참조 파일이 너무 길 경우(300줄 이상), 목차(table of contents)를 포함하세요.

**도메인별 구성:** 하나의 스킬이 여러 도메인/프레임워크를 지원하는 경우, 변형(variant)별로 문서를 구성하세요:
```
cloud-deploy/
├── SKILL.md (작업 흐름 + 어떤 클라우드를 선택할지 판단)
└── references/
    ├── aws.md
    ├── gcp.md
    └── azure.md
```
에이전트가 관련 있는 참조 파일만 읽도록 지시합니다.

#### 놀람 최소화의 원칙 (Principle of Lack of Surprise)

당연한 이야기지만, 스킬에는 악성 코드, 취약점 공격 코드 등 시스템 보안을 위협할 수 있는 어떠한 내용도 포함되어서는 안 됩니다. 스킬의 내용은 설명(description)에서 유추할 수 있는 의도를 벗어나서 사용자를 놀라게 해서는 안 됩니다. 오해를 불러일으키는 스킬이나 무단 접근, 데이터 유출 등을 조장하는 스킬을 만들어 달라는 요청에는 응하지 마세요. (단, "XYZ 역할극을 해줘"와 같은 롤플레잉은 괜찮습니다.)

#### 작성 패턴 (Writing Patterns)

지침을 작성할 때는 가급적 **명령문(imperative form)**을 사용하세요.

**출력 형식 정의하기:** 아래와 같은 패턴을 사용할 수 있습니다.
```markdown
## 보고서 구조
항상 다음의 정확한 템플릿을 사용하십시오:
# [제목]
## 요약
## 주요 발견 사항
## 권고안
```

**예시 패턴:** 예시를 포함하는 것은 매우 유용합니다. 다음과 같이 포맷팅할 수 있습니다 (단, 예시 자체에 "Input"이나 "Output"이라는 단어가 쓰인다면 혼동을 피하기 위해 조금 다르게 표기하세요):
```markdown
## 커밋 메시지 포맷
**예시 1:**
Input: Added user authentication with JWT tokens
Output: feat(auth): implement JWT-based authentication
```

### 작성 스타일 (Writing Style)

에이전트에게 무겁고 낡은 'MUST(반드시 할 것)'를 남발하기보다는, **왜 이것이 중요한지(why)**를 설명하려고 노력하세요. '마음의 이론(theory of mind)'을 활용하고, 특정 예제에만 국한되지 않도록 스킬을 범용적으로 만들어 보세요. 초안을 먼저 작성한 뒤, 새로운 시각으로 다시 검토하고 개선하는 과정이 필요합니다.

### 테스트 케이스 (Test Cases)

스킬 초안을 작성한 후, 2~3개의 현실적인 테스트 프롬프트(실제 사용자가 말할 법한 내용)를 고안해 내세요. 사용자에게 공유하세요: "제가 시도해보고 싶은 테스트 케이스는 다음과 같습니다. 괜찮아 보이시나요, 아니면 더 추가할 내용이 있나요?" (정확히 이 문구를 쓸 필요는 없습니다.) 확인을 받은 후, 평가(실행)를 진행합니다.

테스트 케이스를 `evals/evals.json`에 저장하세요. 아직 검증 기준(assertions)을 작성하지는 마세요 — 오직 프롬프트만 작성합니다. 에발(실행)이 진행되는 동안 다음 단계에서 검증 기준의 초안을 작성할 것입니다.

```json
{
  "skill_name": "example-skill",
  "evals": [
    {
      "id": 1,
      "prompt": "User's task prompt",
      "expected_output": "Description of expected result",
      "files": []
    }
  ]
}
```

(전체 스키마는 `references/schemas.md`를 참고하세요.)

## 2. 테스트 케이스 실행 및 평가 (Evaluating)

테스트 프롬프트(실제 사용자가 할 법한 질문)를 2~3개 작성하여 사용자와 확인한 후 평가를 진행합니다. 서브에이전트를 병렬로 띄울 수 없는 현재 환경을 고려하여 **다음 두 가지 방식 중 하나를 사용자가 선택**하게 합니다.

### 옵션 A: 대화형 순차 테스트 (Interactive Sequential Test)
가장 권장되는 직관적인 방법입니다. 대화창 내에서 에이전트(나)가 직접 시뮬레이션합니다.
1. 사용자가 제시한 테스트 프롬프트를 에이전트가 하나씩 순차적으로 실행해 봅니다.
2. "스킬을 적용하지 않았을 때(Baseline)" 어떻게 행동할지와 "이 스킬을 적용했을 때(With-skill)" 어떻게 결과가 달라지는지 비교하여 산출물을 보여줍니다.
3. 사용자가 결과를 보고 바로 피드백(Inline Feedback)을 줍니다.

### 옵션 B: 자동화 스크립트 기반 테스트 (Automated Script Test)
매번 대화로 테스트하기 번거롭거나 정량적 지표가 필요한 경우에 사용합니다.
1. `evals/evals.json`에 테스트 프롬프트와 예상 결과(Assertions)를 정의합니다.
2. 사용자가 로컬 LLM API (예: Gemini API)를 연동한 Python 테스트 스크립트를 작성하여 여러 케이스를 한 번에 돌리도록 가이드합니다.
3. 결과물은 `{skill-workspace}/iteration-N/` 구조로 저장하고, 제공된 `eval-viewer/generate_review.py`를 실행하여 윈도우 환경(HTML 브라우저)에서 시각적 벤치마크를 확인합니다.
   *실행 예시 (Windows):* `start python .agents/skills/skill-creator/eval-viewer/generate_review.py {workspace-path}`

## 3. 스킬 개선 (Improving the skill)

테스트 결과와 사용자의 피드백을 바탕으로 스킬을 개선합니다.
- **피드백 일반화:** 특정 테스트 케이스에만 과적합(Overfitting)되지 않도록, 일반적인 상황에서도 잘 동작하는지 고민합니다.
- **설명 추가 (Explain the why):** 무조건 `절대 하지 마시오(NEVER)`나 `항상 하시오(ALWAYS)`와 같이 강압적으로 지시하기보다는, "왜 이렇게 해야 하는지" 이유를 설명하여 LLM이 맥락을 이해하도록 돕습니다.
- **반복 작업 추상화:** 테스트 중 여러 번 반복되는 코드 패턴이 있다면 이를 `scripts/` 디렉토리에 파이썬 스크립트 형태로 추출하여 재사용하게 만듭니다.

## 4. 설명 최적화 (Description Optimization)

SKILL.md의 `description`은 이 스킬이 언제 호출될지 결정하는 가장 중요한 요소입니다. CLI 툴 없이 **대화형(Inline) 시뮬레이션**을 통해 트리거 문구를 최적화합니다.

1. **테스트 케이스 생성:** 에이전트는 이 스킬이 "트리거 되어야 하는 상황(Should-trigger)" 10개, "유사하지만 트리거 되면 안 되는 상황(Should-not-trigger)" 10개를 상상하여 총 20개의 가상 사용자 프롬프트를 만듭니다.
2. **시뮬레이션 검토:** 에이전트 스스로 현재 `description`을 읽었을 때, 위 20개의 상황에서 올바르게 트리거(또는 무시) 할 수 있을지 논리적으로 시뮬레이션합니다.
3. **최적화 문구 제안:** 실패가 예상되는 케이스(예: 트리거 되어야 하는데 안 될 것 같거나, 안 되어야 하는데 될 것 같은 상황)를 방어할 수 있도록 `description` 문구를 2~3회 수정해가며 최적의 버전을 찾아 사용자에게 제안합니다.

## 5. 최종 등록 (Publish)

모든 테스트와 설명 최적화가 완료되면, 1번 단계에서 정했던 경로(`.agents/workflows/` 또는 `.agents/skills/`)에 완성된 마크다운 및 파일들을 최종 복사/생성하여 작업을 마무리합니다.
